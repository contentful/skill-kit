Workflow skills are typed state machines. Each step has a prompt, an output schema, and a transition. The agent sees one step at a time — the CLI controls the order, validates outputs, and decides what comes next.

## `skill()` configuration

`skill(config)` returns a `SkillBuilder`. One per entry file. Params and stash Zod schemas declared here flow into all step callbacks via contextual type inference.

```typescript
import { skill, z } from '@contentful/skill-kit';

export default skill({
  name: 'deploy-check',
  version: '1.0.0',
  description: 'Validates deployment readiness across environments.',
  entry: 'gather-info',
  params: z.object({ env: z.string().default('staging') }),
  stash: z.object({ checksPassed: z.number() }),
  finalOutput: z.object({ ready: z.boolean(), report: z.string() }),
  observers: {
    onStepComplete: ({ step, durationMs }) => {
      process.stderr.write(`[deploy-check] ${step} completed in ${durationMs}ms\n`);
    },
  },
  skillMd: 'Custom SKILL.md template override',
})
  .step('gather-info', {
    /* ... */
  })
  .build();
```

| Field                    | Type                            | Required | Description                                                                                                  |
| ------------------------ | ------------------------------- | -------- | ------------------------------------------------------------------------------------------------------------ |
| `name`                   | `string`                        | Yes      | Skill identifier. Used in build output directory name and generated SKILL.md.                                |
| `entry`                  | `string`                        | Yes      | Name of the first step to execute. Must match a registered step name.                                        |
| `version`                | `string`                        | No       | Semver version string. Defaults to `'0.0.0'`.                                                                |
| `description`            | `string`                        | No       | Human-readable description. Included in generated SKILL.md frontmatter.                                      |
| `triggers`               | `string[]`                      | No       | Trigger keywords appended to the description for agent discoverability.                                      |
| `params`                 | `z.ZodType`                     | No       | Zod schema for global skill params. Validated on `start`, typed everywhere.                                  |
| `stash`                  | `z.ZodType`                     | No       | Zod schema for cross-step scratchpad state. Shallow-merged after each step.                                  |
| `finalOutput`            | `z.ZodType`                     | No       | Schema for the skill's "return value". Validated when the workflow terminates.                               |
| `observers`              | `ObserverMap`                   | No       | Read-only lifecycle callbacks for logging, telemetry, and audit.                                             |
| `skillMd`                | `string \| ((skill) => string)` | No       | Custom SKILL.md template. Overrides the generated default.                                                   |
| `argumentHint`           | `string`                        | No       | Autocomplete hint text. Emitted as `argument-hint` in SKILL.md frontmatter.                                  |
| `arguments`              | `string \| string[]`            | No       | Named positional arguments for `$name` substitution in skill content. Emitted as `arguments` in frontmatter. |
| `allowedTools`           | `string \| string[]`            | No       | Pre-approved tools. Emitted as `allowed-tools` in SKILL.md frontmatter.                                      |
| `paths`                  | `string \| string[]`            | No       | Glob patterns for file-based activation. Emitted as `paths` in frontmatter.                                  |
| `context`                | `string`                        | No       | Execution context (e.g. `'fork'`). Emitted as `context` in frontmatter.                                      |
| `license`                | `string`                        | No       | License name or reference. Emitted as `license` in frontmatter.                                              |
| `compatibility`          | `string`                        | No       | Environment requirements. Emitted as `compatibility` in frontmatter.                                         |
| `agent`                  | `string`                        | No       | Subagent type when `context: 'fork'`. Emitted as `agent` in frontmatter.                                     |
| `model`                  | `string`                        | No       | Model override while skill is active. Emitted as `model` in frontmatter.                                     |
| `effort`                 | `string`                        | No       | Effort level override. Emitted as `effort` in frontmatter.                                                   |
| `disableModelInvocation` | `boolean`                       | No       | Prevent auto-loading by the agent. Emitted as `disable-model-invocation`.                                    |
| `userInvocable`          | `boolean`                       | No       | Whether visible in `/` menu. Emitted as `user-invocable`.                                                    |

## `.step(name, config)`

Adds a step to the skill. The prompt and stash callbacks receive typed `params` and `stash` from the builder — no manual annotations needed.

```typescript
.step('gather-info', {
  prompt: ({ params }) => `Check deployment readiness for ${params.env}.`,
  output: z.object({
    checks: z.array(z.object({
      name: z.string(),
      passed: z.boolean(),
      detail: z.string(),
    })),
  }),
  updateStash: ({ stepOutput }) => ({ checksPassed: stepOutput.checks.filter(c => c.passed).length }),
  action: {
    run: writeReportAction,
    input: ({ stepOutput, stash }) => ({ checks: stepOutput.checks }),
    updateStash: ({ actionOutput }) => ({ reportPath: actionOutput.path }),
  },
  next: ({ stepOutput }) => stepOutput.checks.every(c => c.passed) ? 'approve' : 'remediate',
  maxVisits: 3,
  onMaxVisits: 'force-report',
})
```

### Full step config shape

| Field         | Type                                                     | Required | Description                                                              |
| ------------- | -------------------------------------------------------- | -------- | ------------------------------------------------------------------------ |
| `prompt`      | `string \| PromptPiece \| PromptFn`                      | No       | The prose shown to the agent. Can be static, a segment, or a function.   |
| `output`      | `z.ZodType`                                              | Yes      | Zod schema the agent's response must match. Validated before transition. |
| `next`        | `string \| (ctx) => string \| { terminal: true }`        | Yes      | Where to go after this step. Static name, dynamic function, or terminal. |
| `updateStash` | `(ctx: { stepOutput, actionOutput? }) => Partial<Stash>` | No       | Shallow-merge updates into the cross-step scratchpad.                    |
| `action`      | `{ run, input?, updateStash? }`                          | No       | CLI-side side effect that runs after output validation.                  |
| `maxVisits`   | `number`                                                 | No       | Explicit visit limit for this step.                                      |
| `onMaxVisits` | `string`                                                 | No       | Fallback step when `maxVisits` is exceeded.                              |

## PromptContext

Every dynamic `prompt` function receives a `PromptContext` object. These are the fields available:

| Field      | Type                                                           | Description                                                                     |
| ---------- | -------------------------------------------------------------- | ------------------------------------------------------------------------------- |
| `history`  | `readonly StepResult[]`                                        | All prior step results as `{ step, stepOutput, actionOutput? }` objects.        |
| `getStep`  | `(name: string) => { stepOutput; actionOutput? } \| undefined` | Type-safe lookup of a specific step's result from history (typed from builder). |
| `params`   | `TParams`                                                      | Global skill params (config defaults + runtime overrides).                      |
| `refs`     | `ReferenceLoader`                                              | Lazy loader for reference files: `refs.load('file.md')`, `refs.asset('path')`.  |
| `attempts` | `number`                                                       | How many times this step has been visited (zero-indexed, useful for retries).   |
| `host`     | `Handshake`                                                    | Host info: `{ host: string, toolsAvailable: string[] }`.                        |
| `stash`    | `Readonly<TStash>`                                             | Current cross-step scratchpad state. Frozen — read-only in prompts.             |
| `act`      | `ActContext`                                                   | Primitive directive builders: `askUser`, `confirm`, `plan`, etc.                |
| `system`   | `SystemTag`                                                    | System segment tag/function for persona/frame.                                  |

## Transitions

Every step declares a `next` that determines where the workflow goes after the agent responds.

### Static transition

A fixed step name. The simplest case.

```typescript
.step('gather', {
  output: GatherSchema,
  next: 'analyze',
})
```

### Dynamic transition

A function that receives the step output and returns a step name. Use this for branching logic.

```typescript
.step('diagnose', {
  output: z.object({ severity: z.enum(['low', 'medium', 'high']) }),
  next: ({ stepOutput }) => stepOutput.severity === 'high' ? 'escalate' : 'auto-fix',
})
```

The function receives `{ stepOutput, attempts, actionOutput }` where `actionOutput` is the action result if one ran.

### Terminal transition

Ends the workflow. The last step's output becomes the skill's final output.

```typescript
.step('report', {
  output: ReportSchema,
  next: { terminal: true },
})
```

### Self-transition

Return `'self'` from a dynamic `next` to revisit the current step. Always pair with a loop guard.

```typescript
.step('refine', {
  output: z.object({ confidence: z.number(), result: z.string() }),
  next: ({ stepOutput, attempts }) =>
    stepOutput.confidence < 0.8 && attempts < 3 ? 'self' : 'done',
  maxVisits: 3,
  onMaxVisits: 'done',
})
```

## Loop guards

Cycles are detected by static graph analysis at build time. Any step that participates in a cycle gets an implicit visit limit of **10**. For explicit control, declare `maxVisits` and `onMaxVisits`.

| Config                                   | Behavior                                                             |
| ---------------------------------------- | -------------------------------------------------------------------- |
| No `maxVisits` declared, step in a cycle | Implicit limit of 10 visits. Throws `CycleGuardError` when exceeded. |
| `maxVisits` without `onMaxVisits`        | Throws `CycleGuardError` when the limit is hit (fail-closed).        |
| `maxVisits` + `onMaxVisits`              | Redirects to the `onMaxVisits` step when the limit is hit.           |

Linear workflows with function-based `next` don't need cycle guards — the conservative graph analysis handles safety automatically by treating dynamic transitions as potentially reaching all steps.

```typescript
.step('retry-step', {
  output: RetrySchema,
  next: ({ stepOutput }) => stepOutput.ok ? 'done' : 'self',
  maxVisits: 5,
  onMaxVisits: 'fallback',  // required safety valve
})
```

## Stash merging

The stash is a cross-step scratchpad typed by the skill's `stash` schema. Updates are **shallow-merged** — each `updateStash` callback returns a partial object that overwrites top-level keys.

```typescript
// Step 1 sets name
updateStash: ({ stepOutput }) => ({ name: stepOutput.name }),
// Stash is now { name: 'Alice' }

// Step 2 sets role — name is preserved
updateStash: ({ stepOutput }) => ({ role: stepOutput.role }),
// Stash is now { name: 'Alice', role: 'dev' }
```

The stash exposed in prompt functions is a **frozen** (read-only) snapshot. Mutations are only possible through `updateStash` and `action.updateStash` callbacks — never inside prompts.

## `.extend(name, base, overrides)`

Wires a shared step into the skill with typed overrides. The base step provides defaults; overrides replace specific fields. Params and stash types from the builder flow into the override callbacks.

```typescript
import { step, z } from '@contentful/skill-kit';

// Shared step — no params/stash types
const openQuestion = step({
  output: z.object({ answer: z.string() }),
  next: '__parent__',
});

// In a skill — .extend() applies typed params/stash:
skill({ params: ParamsSchema, stash: StashSchema, entry: 'ask', ... })
  .extend('ask', openQuestion, {
    prompt: ({ stash }) => `${stash.name}, tell me about yourself.`,  // typed
    next: 'done',  // overrides __parent__
  })
```

## `.build()` validation

Calling `.build()` freezes the skill definition and validates:

- `name` is present
- `entry` is present and matches a registered step
- At least one step exists
- No unresolved `__parent__` sentinels remain (they must be overridden via `.extend()` or `.register()`)

Triggers, if present, are appended to the description string for agent discoverability.

## Handling validation errors

When the agent's output doesn't match a step's Zod schema, the engine:

1. Fires `onStepValidationFailed` with the raw output, error message, and attempt number
2. Returns a `ValidationErrorResult` to the agent with `retry: true`
3. The agent sees the error message and tries again

The skill author doesn't need to handle this — the engine manages the retry loop automatically. But the `onStepValidationFailed` observer is invaluable during development:

```typescript
skill({
  // ...
  observers: {
    onStepValidationFailed: ({ step, raw, error, attempt }) => {
      console.error(`Step "${step}" attempt ${attempt} failed validation: ${error}`);
      console.error('Raw output:', JSON.stringify(raw, null, 2));
    },
  },
});
```

### Testing validation paths

Use a function response in `mockModel` to return invalid output on the first attempt, then valid output on the retry:

```typescript
import { runSkill, mockModel } from '@contentful/skill-kit/test';

const result = await runSkill(mySkill.build(), {
  model: mockModel({
    'my-step': (prompt) => {
      if (prompt.includes('validation')) {
        return { valid: 'output' };
      }
      return { bad: 'shape' }; // first call — triggers validation error + retry
    },
  }),
});
```

## Standalone `step()` function

For shared, reusable steps defined outside a skill. These steps have no typed params or stash — they default to `unknown`. Use `__parent__` as the next transition, then override it with `.extend()`.

```typescript
import { step, z } from '@contentful/skill-kit';

export const gatherRepoFacts = step({
  prompt: 'Inspect the repo and list languages, build tools, CI config.',
  output: z.object({
    languages: z.array(z.string()),
    buildTools: z.array(z.string()),
    ci: z.string().nullable(),
  }),
  next: '__parent__',
});
```

The standalone `step()` validates that `output` and `next` are provided. The returned `StepDefinition` is frozen and has an `.extend(overrides)` method for inline customization.

## Complete worked example

A deploy-check skill that gathers environment info, presents a plan, asks the user which checks to run, tracks remediation tasks, and produces a report. Demonstrates `act.askUser`, `act.plan`, `act.checklist`, `act.confirm`, fragments, stash, branching, and loop guards.

```typescript
import { skill, step, z, prompt, fragment, render, act } from '@contentful/skill-kit';

// --- Fragments ---

const strictTone = fragment(
  'strict-tone',
  `Be precise and thorough. No hand-waving. Every claim must be backed
   by a specific file path, command output, or config value.`,
);

// --- Schemas ---

const CheckResult = z.object({
  name: z.string(),
  passed: z.boolean(),
  detail: z.string(),
});

// --- Skill ---

export default skill({
  name: 'deploy-check',
  version: '1.0.0',
  description: 'Validates deployment readiness across environments.',
  entry: 'choose-env',

  params: z.object({
    defaultEnv: z.string().default('staging'),
  }),

  stash: z.object({
    env: z.string(),
    failCount: z.number(),
  }),

  finalOutput: z.object({
    ready: z.boolean(),
    summary: z.string(),
  }),
})
  // Step 1: Ask the user which environment to check
  .step('choose-env', {
    prompt: act.askUser({
      type: 'structured',
      question: 'Which environment should we validate?',
      options: [
        { value: 'staging', label: 'Staging' },
        { value: 'production', label: 'Production', description: 'Live environment' },
        { value: 'local', label: 'Local dev' },
      ],
    }),
    output: z.object({ env: z.enum(['staging', 'production', 'local']) }),
    updateStash: ({ stepOutput }) => ({ env: stepOutput.env }),
    next: 'plan-checks',
  })

  // Step 2: Present the check plan for approval
  .step('plan-checks', {
    prompt: act.plan({
      summary: 'Deployment readiness checks',
      steps: [
        'Verify the build completes without errors',
        'Run the full test suite',
        'Confirm environment variables are set',
        'Validate database migrations are current',
      ],
    }),
    output: z.object({ approved: z.boolean() }),
    next: ({ stepOutput }) => (stepOutput.approved ? 'run-checks' : 'force-report'),
  })

  // Step 3: Run the checks
  .step('run-checks', {
    prompt: ({ stash }) => prompt`
      ${strictTone}

      Run deployment checks for the **${stash.env}** environment:
      1. Verify the build completes without errors
      2. Check that all tests pass
      3. Confirm environment variables are set
      4. Validate database migrations are current

      Return each check with its pass/fail status and detail.
    `,
    output: z.object({ checks: z.array(CheckResult) }),
    updateStash: ({ stepOutput }) => ({
      failCount: stepOutput.checks.filter((c) => !c.passed).length,
    }),
    next: ({ stepOutput }) => (stepOutput.checks.every((c) => c.passed) ? 'approve' : 'remediate'),
  })

  // Step 4a: Remediation (if checks failed) — uses act.checklist to track fixes
  .step('remediate', {
    prompt: ({ getStep, stash, act }) => [
      act.checklist({
        create: [
          { title: 'Fix failing deployment checks', status: 'in_progress' },
          { title: 'Re-run validation suite', status: 'pending' },
        ],
      }),
      prompt`
        ${strictTone}

        ${stash.failCount} check(s) failed for **${stash.env}**.
        Failed checks: ${JSON.stringify(getStep('run-checks')?.stepOutput, null, 2)}

        For each failure, propose a concrete fix. Then re-run the checks.
      `,
    ],
    output: z.object({ checks: z.array(CheckResult) }),
    updateStash: ({ stepOutput }) => ({
      failCount: stepOutput.checks.filter((c) => !c.passed).length,
    }),
    next: ({ stepOutput, attempts }) => {
      if (stepOutput.checks.every((c) => c.passed)) return 'approve';
      if (attempts >= 2) return 'force-report';
      return 'self';
    },
    maxVisits: 3,
    onMaxVisits: 'force-report',
  })

  // Step 4b: Approval (if all checks passed)
  .step('approve', {
    prompt: act.confirm({
      message: 'All checks passed. Ready to generate the deployment report?',
      destructive: false,
      defaultAnswer: 'yes',
    }),
    output: z.object({ approved: z.boolean() }),
    next: ({ stepOutput }) => (stepOutput.approved ? 'report' : 'run-checks'),
  })

  // Step 5a: Final report (happy path)
  .step('report', {
    prompt: ({ history, stash }) => {
      const lastChecks = history.filter((s) => s.step === 'run-checks' || s.step === 'remediate').pop();
      const checks =
        (lastChecks?.stepOutput as { checks: Array<{ name: string; passed: boolean; detail: string }> })?.checks ?? [];
      const table = render.table(checks, { columns: ['name', 'passed', 'detail'] });
      const report = render.section(`Deployment Report: ${stash.env}`, table);
      return `Output the following deployment report exactly as shown:\n\n${report}`;
    },
    output: z.object({ ready: z.boolean(), summary: z.string() }),
    next: { terminal: true },
  })

  // Step 5b: Force report (after max remediation attempts)
  .step('force-report', {
    prompt: ({ stash }) => prompt`
      Remediation attempts exhausted for **${stash.env}**.
      Generate a summary noting which checks still fail and recommend manual intervention.
    `,
    output: z.object({ ready: z.boolean(), summary: z.string() }),
    next: { terminal: true },
  })

  .build();
```

### Testing the deploy-check skill

```typescript
import test from 'node:test';
import assert from 'node:assert/strict';
import { runSkill, mockModel } from '@contentful/skill-kit/test';
import skill from './skill.js';

test('happy path: all checks pass', async () => {
  const result = await runSkill(skill, {
    params: { defaultEnv: 'staging' },
    model: mockModel({
      'choose-env': { env: 'staging' },
      'plan-checks': { approved: true },
      'run-checks': {
        checks: [
          { name: 'build', passed: true, detail: 'Build succeeded' },
          { name: 'tests', passed: true, detail: '142 tests passed' },
        ],
      },
      approve: { approved: true },
      report: { ready: true, summary: 'All checks passed for staging.' },
    }),
  });

  assert.deepEqual(result.path, ['choose-env', 'plan-checks', 'run-checks', 'approve', 'report']);
  assert.equal((result.output as { ready: boolean }).ready, true);
});

test('remediation path: failed checks trigger fixes', async () => {
  const result = await runSkill(skill, {
    model: mockModel({
      'choose-env': { env: 'production' },
      'plan-checks': { approved: true },
      'run-checks': {
        checks: [
          { name: 'build', passed: true, detail: 'OK' },
          { name: 'migrations', passed: false, detail: 'Pending migration' },
        ],
      },
      remediate: {
        checks: [
          { name: 'build', passed: true, detail: 'OK' },
          { name: 'migrations', passed: true, detail: 'Migration applied' },
        ],
      },
      approve: { approved: true },
      report: { ready: true, summary: 'Fixed and ready.' },
    }),
  });

  assert.ok(result.path.includes('remediate'));
  assert.ok(result.path.includes('approve'));
});
```
